# 维度 6：校验规则与异常处理（Validation & Exception Handling）

> 评估 PRD 是否覆盖了所有校验规则、异常路径与兜底机制。校验规则缺失 = 上线后必出资金/数据事故。

**权重：12%**

---

## 评估问题清单（共 13 题）

### P0（致命）

| ID | 问题 | 判定标准 | 命中证据示例 |
|---|---|---|---|
| **D6-Q1** | 是否列出了所有的业务校验规则？ | 校验规则集中列出，含触发条件 | 缺失：仅写"系统校验"无具体规则 |
| **D6-Q2** | 校验失败时是否给出具体的错误提示文案？ | 每条校验对应一句完整提示文案 | 缺失：仅写"校验失败"无提示文 |
| **D6-Q3** | 是否区分了「前端校验」和「后端校验」（含 OA / 系统级）？ | 校验分层清晰，避免重复或遗漏 | 缺失：所有校验混在一起 |

### P1（重要）

| ID | 问题 | 判定标准 | 命中证据示例 |
|---|---|---|---|
| **D6-Q4** | 是否覆盖了边界场景（空值 / 极值 / 重复值 / 并发）？ | 显式说明"借据号不能重复"、"金额不能为 0"等 | 缺失：未考虑空值、重复、极值 |
| **D6-Q5** | 是否说明了时间窗口约束？ | 时间相关业务有窗口规则（如「距离截止时间 < 4h 不允许提交」） | 缺失：时间敏感操作无窗口约束 |
| **D6-Q6** | 是否定义了兜底机制 / 紧急关闭开关？ | 显式说明"研发预留口子可关闭该校验" | 缺失：临时 case 无逃生通道 |
| **D6-Q7** | 是否说明了校验顺序（哪些先校验、命中后是否跳过）？ | 校验之间的先后关系明确 | 缺失：多校验同时报错难以定位 |
| **D6-Q8** | 是否定义了校验通过 / 失败后的下一步动作？ | 通过后自动审批 / 失败后驳回 / 失败后通知申请人 | 缺失：校验失败后无后续动作 |
| **D6-Q9** | 是否说明了错误的恢复路径（重试 / 回滚 / 驳回 / 修正后重提）？ | 用户/运营拿到错误后知道下一步怎么做 | 缺失：失败后无指引 |

### P2（建议）

| ID | 问题 | 判定标准 | 命中证据示例 |
|---|---|---|---|
| **D6-Q10** | 错误提示文案是否对用户友好（指引下一步动作）？ | 提示包含原因 + 修复建议（如"请删除重复借据号"） | 命中：仅"操作失败"无指引 |
| **D6-Q11** | 是否说明校验结果的留痕方式（OA 附件 / 操作记录 / 日志）？ | 校验结果可被回查 | 缺失：失败原因无留痕，无法追溯 |
| **D6-Q12** | 是否区分了「警告」（可继续）和「错误」（必须阻断）？ | 校验结果分级，部分允许用户确认后继续 | 缺失：所有校验都是阻断级 |
| **D6-Q13** | 是否考虑了校验性能（大数据量）？ | 大量级（如 53 万笔）校验有时效说明、异步方案 | 缺失：大数据量校验阻塞用户 |

---

## 评分要点

- 本维度是金融/账务类 PRD 的「保命底线」，每条 P0 命中都应在报告"风险提示"中单独高亮。
- **D6-Q2（提示文案）** 是优秀 PRD 的强标识：参考《资产债转平台一期》中每条校验都附上提示文案如"借据号重复请删除"。
- **D6-Q5（时间窗口）** 在审批/调度类业务中极易遗漏：如未定义「距离债转时间 [0,4h) 不允许提交」会导致研发临时返工。
- **D6-Q6（兜底开关）** 是务实派 PRD 的标志：临时 case 必须有逃生口，避免线上事故。
- **D6-Q13（性能）** 在数据量级 ≥ 10 万笔时必须考虑，否则一条同步校验就能让页面崩溃。
