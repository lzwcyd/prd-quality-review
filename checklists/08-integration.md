# 维度 8：系统集成与上下游（Integration & Dependencies）

> 评估 PRD 是否说清了"涉及哪些系统、谁负责什么、接口怎么对、消息怎么传"。集成不清是跨团队联调失败的首因。

**权重：10%**

---

## 评估问题清单（共 13 题）

### P0（致命）

| ID | 问题 | 判定标准 | 命中证据示例 |
|---|---|---|---|
| **D8-Q1** | 是否列出了所有相关上下游系统？ | 显式列出涉及系统（账务 / 客账 / 数仓 / 合同 / CIS 等） | 缺失：跨多系统但只提"系统对接" |
| **D8-Q2** | 是否明确了各系统的职责分工（谁做什么）？ | 责任矩阵 / 表格清晰，无重叠或遗漏 | 缺失：职责不清导致互相推 |
| **D8-Q3** | 是否给出了系统间的接口字段定义？ | 接口/消息字段含名称、类型、含义 | 缺失：仅说"提供接口"无字段 |

### P1（重要）

| ID | 问题 | 判定标准 | 命中证据示例 |
|---|---|---|---|
| **D8-Q4** | 是否说明了交互方式（同步接口 / MQ 消息 / 定时任务 / Webhook）？ | 每个交互显式标注交互方式 | 缺失：交互方式靠猜 |
| **D8-Q5** | 是否定义了消息体 / 接口入参 / 出参字段？ | 入参出参字段清单完整 | 缺失：仅有接口名无字段 |
| **D8-Q6** | 是否说明了消息时序与幂等性？ | 异步消息说明消费幂等、重复消费如何处理 | 缺失：MQ 消费无幂等设计 |
| **D8-Q7** | 是否标注了相关产品 / 研发负责人？ | 每个系统对接有 owner 指派 | 缺失：跨团队但无对接人 |
| **D8-Q8** | 是否说明了改动点是「新增」还是「改造」？ | 显式区分新接口 vs 旧接口改造 | 缺失：旧接口改动未告知下游 |
| **D8-Q9** | 是否考虑了下游系统的影响面？ | 显式说明"重点排查 PGC、短信、特征平台等"等下游 | 缺失：未盘点下游消费方 |

### P2（建议）

| ID | 问题 | 判定标准 | 命中证据示例 |
|---|---|---|---|
| **D8-Q10** | 是否说明了接口的 SLA（响应时间 / 成功率 / 重试策略）？ | 标注接口 SLA 要求 | 缺失：性能预期不清 |
| **D8-Q11** | 是否说明了灰度切换 / 降级方案？ | 接口有灰度开关和降级路径 | 缺失：新接口无灰度切换 |
| **D8-Q12** | 是否说明了接口的旧版本兼容 / 废弃计划？ | 旧接口何时下线、如何兼容 | 缺失：新旧接口长期共存无规划 |
| **D8-Q13** | 是否列出了所有需要联调对接的相关方与文档？ | 文末或附录列联调清单 | 缺失：联调时才发现遗漏方 |

---

## 评分要点

- **D8-Q1 + D8-Q2** 是金融系统的灵魂：参考案例《资产债转凭证》在「需求相关方」表中清晰列出资产债权/账务模块、监管系统、数仓、资金模块、合同模块、客服系统的负责产品和改动点，是上下游协作的范本。
- **D8-Q9（下游影响面）** 极易遗漏：参考案例《不良资产债转-账务侧》明确写"需评估下所有查询还款计划的下游，重点排查下 PGC 及短信"，这种主动盘点是优秀 PRD 的标志。
- **D8-Q6（消息幂等）** 在异步消息架构下是高发漏点；任何 MQ 消息相关 PRD 缺失幂等说明都应高亮。
- 对纯前端 / 单系统 PRD，本维度多数问题可判 N/A。
