# 维度 7：数据与历史兼容（Data & Backward Compatibility）

> 评估 PRD 是否说清了"老数据怎么办"、"灰度怎么放"、"出问题怎么回滚"。历史兼容缺失会导致上线即数据混乱。

**权重：10%**

---

## 评估问题清单（共 13 题）

### P0（致命）

| ID | 问题 | 判定标准 | 命中证据示例 |
|---|---|---|---|
| **D7-Q1** | 是否明确了历史数据如何处理（补齐 / 迁移 / 不处理）？ | 显式说明历史数据策略 | 缺失：仅做新需求未提历史数据 |
| **D7-Q2** | 是否说明了历史数据迁移的执行方式（手动 / 技术刷数 / 自动）？ | 说明谁来执行、什么方式 | 缺失：仅说"补齐"无执行方案 |
| **D7-Q3** | 是否给出了存量与增量的处理差异？ | 存量与增量分别说明，避免共用逻辑出错 | 缺失：存量逻辑未单列 |

### P1（重要）

| ID | 问题 | 判定标准 | 命中证据示例 |
|---|---|---|---|
| **D7-Q4** | 是否列出了需要补齐的历史记录清单？ | 历史记录有具体清单（数量、时间范围） | 缺失：仅说"补齐历史"无清单 |
| **D7-Q5** | 是否说明了历史数据校验 / 对账方案？ | 补齐后有数据正确性验证机制 | 缺失：补完无验证，无法保证正确 |
| **D7-Q6** | 是否说明了灰度发布策略（按子资金 / 按机构 / 按时间）？ | 显式说明灰度维度与节奏 | 缺失：金融类需求一刀切上线 |
| **D7-Q7** | 是否考虑了数据回滚方案？ | 出问题时如何回退到上线前状态 | 缺失：刷数失败无回滚预案 |
| **D7-Q8** | 是否说明了数据兼容性（旧字段如何兼容、是否新增字段）？ | 新增字段对旧数据的处理方案明确 | 缺失：新字段无默认值导致空指针 |
| **D7-Q9** | 是否说明了历史数据的时间分界点？ | 显式说明 "X 日前 / 后" 数据走不同逻辑（如「25.1.1 前切资管的项目」） | 缺失：分界不清导致逻辑混乱 |

### P2（建议）

| ID | 问题 | 判定标准 | 命中证据示例 |
|---|---|---|---|
| **D7-Q10** | 是否说明了数据归档 / 清理策略？ | 长期数据有归档/清理周期 | 缺失：数据无限增长 |
| **D7-Q11** | 是否说明了字段级别的默认值 / 空值处理？ | 关键字段空值时如何处理（默认值 / 报错 / 跳过） | 缺失：空值处理靠研发猜 |
| **D7-Q12** | 是否考虑了字段类型变更 / 枚举值新增的兼容方案？ | 新枚举值对旧消费方的影响有说明 | 缺失：新枚举导致下游解析失败 |
| **D7-Q13** | 是否提供了数据迁移的责任人与时间表？ | 标注负责人与执行节点 | 缺失：补数任务无 owner |

## 评分要点

- 本维度在涉及"老系统改造 / 替换原有方案"的 PRD 中权重应上调；纯新建需求权重可下调。
- **D7-Q4（历史清单）** 优秀案例《资产债转平台一期》专门有"3.3 历史 3 次债转记录和凭证补齐"章节，列出 4 次债转的封包时间、债转时间、清单链接、校验内容、校验结果，是历史数据章节的典范。
- **D7-Q6（灰度）** 在金融类需求中是必须的：参考案例《资产债转凭证》明确说"按资方（子资金）开启的灰度过程"。
- **D7-Q7（回滚）** 是 PRD 普遍盲点：50% 以上的 PRD 都未提回滚方案，应在评审时强力补齐。
