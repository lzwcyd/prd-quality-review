# 维度 1：基本信息与变更管理（Metadata & Change Control）

> 评估 PRD 的"身份证"信息是否完整、变更过程是否可追溯。这是 PRD 进入研发流程的最低门槛。

**权重：8%**

---

## 评估问题清单（共 11 题）

### P0（致命）

| ID | 问题 | 判定标准 | 命中证据示例 |
|---|---|---|---|
| **D1-Q1** | 是否有可追溯的需求标识（云效 / Jira / 任务编号或链接）？ | PRD 顶部或显著位置有可点击的需求管理系统链接 | 缺失：无任何任务追踪链接 |
| **D1-Q2** | PRD 标题是否明确描述需求范围（动作 + 对象 + 范围）？ | 标题能让人 5 秒内判断"做什么"。模糊标题如"系统优化"判命中 | 缺失：标题为"功能改造" |
| **D1-Q3** | 是否标注了产品负责人 / 作者？ | 出现作者署名或负责人信息（也可在操作记录中体现） | 缺失：无任何责任人标识 |

### P1（重要）

| ID | 问题 | 判定标准 | 命中证据示例 |
|---|---|---|---|
| **D1-Q4** | 是否关联了上游文档（BRD / MRD / 会议纪要 / 关联需求）？ | 文档开头列出 BRD 链接 或 关联需求链接 | 缺失：复杂业务无任何上游引用 |
| **D1-Q5** | 是否有操作记录表（变更时间 / 操作人 / 操作内容）？ | 文档开头存在表格记录每次修改 | 缺失：PRD 已多版迭代但无变更记录 |
| **D1-Q6** | 当 PRD 有变更时，每次变更是否说明变更原因？ | 操作记录的"操作内容"列说明 What + Why，而非仅 What | 命中：仅写"修改了 X 字段"，未说明原因 |
| **D1-Q7** | 是否标注 PRD 版本号或最后更新时间？ | 顶部明确标注版本号 / 最后更新日期 | 缺失：无任何版本/日期标识 |

### P2（建议）

| ID | 问题 | 判定标准 | 命中证据示例 |
|---|---|---|---|
| **D1-Q8** | 是否标注期望上线时间 / 关键里程碑？ | 文档中显式提到"期望上线时间：YYYY-MM-DD"或排期表 | 缺失：未声明任何时间预期 |
| **D1-Q9** | 是否列出了主要 reviewer / 干系人？ | 列出参与评审的产品/研发/测试/业务方 | 缺失：跨多系统但无干系人列表 |
| **D1-Q10** | 标题命名是否遵循团队规范（如【PRD】前缀、模块前缀）？ | 与同团队既有 PRD 命名风格一致 | 命中：团队规范要求【PRD】前缀但未加 |
| **D1-Q11** | 是否在文档开头提供一句话摘要（abstract）？ | 100 字以内说明"做什么 + 为什么"，便于快速浏览 | 缺失：长 PRD 直接进入背景章节 |

---

## 评分要点

- **D1-Q1（云效链接）** 是最常被忽略但又最影响协作效率的项；金融/账务类 PRD 必须有需求追踪。
- **D1-Q5（操作记录表）** 是优秀 PRD 的标志性特征：所有 5 份参考案例都有该表，建议作为团队"PRD 模板"硬性要求。
- 同一作者多次微调但每条只写一行内容，说明变更控制薄弱，应在 D1-Q6 命中。
