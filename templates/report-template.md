## 三-A、⚠️ PRD-代码冲突清单（仅在 CODE_SCAN 启用且有冲突时出现）

> **以下冲突已强制升级为 P0**。冲突意味着 PRD 描述与现有代码实现不一致，会导致研发疑惑或线上事故。

### 🔥 CONFLICT-01 [D5-Q4] 字段类型不一致
- **冲突维度**：5 - 字段类型/枚举
- **PRD 描述**：第 3.2 节定义"金额：integer，单位元"
- **代码现状**：`PaymentDTO.java:L18` 字段类型为 `BigDecimal amount`（精度 2 位，单位元）
- **影响**：若按 PRD 实现会丢失小数精度，引发资金计算错误
- **对齐方向**：
  - 方向 A（修 PRD）：将 PRD 改为 "BigDecimal，精度 2 位"
  - 方向 B（改代码）：与现有支付链路统一使用 integer（不推荐，破坏一致性）
- **建议方向**：A（保持代码不动，修 PRD）

### 🔥 CONFLICT-02 [D5-Q4] 枚举值不一致
- **冲突维度**：5 - 字段类型/枚举
- **PRD 描述**：债转模式枚举为 "双池/原状/代偿/税务/出售"
- **代码现状**：`enum TransferMode { DUAL_POOL, ORIGINAL, COMPENSATE, REPURCHASE, TAX, SALE_OUT }`（共 6 项）
- **影响**：PRD 漏写"回购"枚举值，研发可能误删
- **对齐方向**：建议修 PRD 补齐"回购"

## 七、亮点（值得保留与推广）

> 评估时不只挑刺，也要识别 PRD 中的亮点，形成团队 PRD 范式沉淀。

1. **操作记录表完整**（D1）：详尽记录了每次变更的时间、操作人、变更原因，便于追溯。
2. **业务流程时序图清晰**（D4）：用 Mermaid 描绘 5 个系统协作，是金融跨系统 PRD 的标杆。
3. **校验规则带提示文案**（D6）：每条校验规则后跟具体提示文案（如"借据号重复请删除"），用户体验佳。
4. **PRD-代码命名一致**（代码佐证）：核心字段命名与 model 完全一致，研发可直接对照实现。
